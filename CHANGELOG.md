## 2026-05-07

- Finish the canonical-path plumbing for the watcher fix. `prepare_workspace_state` already canonicalized `workspace_root`, but `build_restore_bundle` and the frontend's `openWorkspace` were still calling `read_directory` with the user's raw input, so child entry paths landed in `directoryCache` under an un-canonical key while the watcher fired canonical paths. Also: the watcher now walks every `.md` descendant when a folder enters the tree via Create(Folder) or Modify(Name) (rename-in), since FSEvents on macOS doesn't re-emit per-child Create events for a renamed inode — without the walk, search and the sidebar's directory-has-markdown indicator would silently drift from disk on rename. `is_workspace_ignored` now consumes the event-kind-derived `is_dir` instead of the unreliable `path.is_dir()` (which returns false on a deleted path), so dir-only gitignore rules like `dist/` correctly match removed directories.
- Reflect Finder "Move to Trash" and `mv` renames in the sidebar. macOS surfaces these as `Modify(Name(_))` rather than `Remove(_)` — the previous fix only handled Create/Remove, so trashing a file (the most common way users delete) didn't update the UI. The watcher now treats Modify(Name) as a directory-membership change and uses `path.exists()` as ground truth: vanished `.md` paths get pruned from the index, vanished non-`.md` paths trigger a subtree prune in case it was a renamed-away folder, and the parent's `fs:directory-changed` fires either way.
- Reflect external deletes and non-markdown changes in the sidebar. Follow-up to today's external-file-watcher fix: the watcher only emitted `fs:directory-changed` for the parent when an `.md` file was created or removed, so deleting a `.txt`, image, or any folder produced no UI refresh. Also `path.is_dir()` is unreliable on a deleted path (the file is already gone by the time we read the event), so folder deletes that came in as `Remove(File)` rather than `Remove(Folder)` were misclassified. The event-handling block is consolidated: `is_dir` now trusts the FSEvent kind first and falls back to a live stat, every Create/Remove fires a parent `fs:directory-changed` regardless of file type, and a removed folder also drops its descendants from the file index in case FSEvents coalesced their per-child events into a single batch. The `Create` branch now checks `path.exists()` before adding to the index so coalesced Create+Remove pairs from FSEvents don't leak phantom entries.
- Pick up external file and folder changes reliably. Three fixes in `apps/desktop/src-tauri/src/watcher.rs` and `commands/workspace.rs`: (1) `should_ignore` now strips the workspace root before scanning for dot-prefixed components, so a workspace at e.g. `~/.notes/` keeps firing events instead of having every path filtered. (2) `prepare_workspace_state` canonicalizes the workspace root on open, so FSEvents-emitted paths (which are always canonical — macOS aliases `/var → /private/var`, plus user symlinks) line up with the frontend's stored root and the `path === root` check in the file watcher hook actually matches. (3) `is_self_write` now uses a TTL window instead of consuming the entry on first match — a single Writer save fans out into Create + Modify(Metadata) + Modify(Data) on FSEvents, and the previous logic only suppressed the first one, letting the others bounce back as `fs:file-changed` events that could clobber in-progress edits. Embeds [Choochmeque/tauri-webdriver](https://github.com/Choochmeque/tauri-webdriver) behind a new `e2e` Cargo feature so the WebDriver server is never compiled into release builds. Run with `pnpm run test:e2e` from `apps/desktop/e2e/` after a one-time `cargo install tauri-webdriver --locked`. The build uses an isolated bundle identifier (`com.writer-computer.e2e`) so it does not conflict with `tauri-plugin-single-instance` from a Writer dev/release instance in another worktree. Two smoke specs: (1) asserts the React app mounts via WebdriverIO + WKWebView, (2) creates a temp file via real `create_file` + `write_file` IPC calls driven from the WKWebView's JS context, and verifies the bytes hit disk. Official `tauri-driver` still does not support macOS (tauri-apps/tauri#7068).
- Improved cold-start performance. Collapsed the startup IPC waterfall (previously six sequential round-trips: `get_startup_state` → `file_exists` → `open_workspace` → `read_directory` → `get_recent_workspaces` → `load_session` → `read_file`) into a single `get_startup_state` call. The Rust command is now `async` and, when `window.restore-workspace` is enabled and the most-recent recent still exists on disk, prefetches the full workspace restore bundle — directory entries, recents, session, and the active tab's file content — via a shared `build_restore_bundle` helper that fans out `read_directory_impl` / `load_recent_workspaces` / `load_session_impl` over `spawn_blocking` and then reads the active file. The frontend's `resolveStartup` hydrates `useSettingsStore` and `useWorkspaceStore` synchronously from the response _before_ flipping `isStartupResolved`, so React's very first render paints the full editor with content already in place — no second IPC, no intermediate blank frame. `editor-store.restoreSession` accepts an optional `prefetchedActiveFile` so the active tab skips its own `readFile` round-trip and the `OPEN_FILE_GRACE_MS` wait. The new `restore_workspace` Tauri command stays registered so user-initiated workspace switches can reuse the same bundled path (same `build_restore_bundle` under the hood). Added a dev-only `lib/startup-metrics.ts` helper that drops `performance.mark()` calls at the entry, around `get_startup_state`, at `setStartupResolved`, and at first CodeMirror view creation, dumping a `console.table` timeline so regressions are visible in DevTools. Removed the now-dead `resolveStartupAction` decision oracle and its tests since the path is decided directly from the restore bundle / `pending_open`.
- Removed the 75 ms `animate-fade-in` opacity cycle on the app root (the "blank window" it produced at cold start was the entire motivation for the fade). Dropped the `@utility animate-fade-in` + `@keyframes fade-in` from `App.css` and the `className` from both branches in `App.tsx`. The window now appears with the editor already rendered at full opacity because `resolveStartup` flips `isStartupResolved` only after the bundled `get_startup_state` IPC has hydrated the stores.
- `restoreFromBundle` in `workspace-store.ts` now fires `restoreSession` as a `void` (fire-and-forget) with a `.catch(console.error)` safety net. The synchronous `set()` inside `restoreSession` still populates the active tab from `prefetchedActiveFile` before yielding, so the first render has content; background tab reads that used to block `setStartupResolved` now run in the background and fill in their tabs as they land.
- Reduced document open latency (`SPECs/document-open-latency-spec.md`). Cached `displayDate` and `stats` on `OpenFile` so `DocumentHeader` / `DocumentFooter` no longer re-parse frontmatter and re-scan content on every render, exposed via new `useFileDisplayDate` / `useFileStats` selectors. Shrunk `INITIAL_PARSE_BUDGET_MS` from 75 ms to 20 ms and doubled `MAX_FOLLOW_UP_PARSE_FRAMES` (4 → 8) so CodeMirror's parse budget is preserved but distributed across idle frames instead of blocking the first keystroke after a file open. Added a 40 ms grace window in `openFile` that races `ensureFileLoaded` against a timer before creating the tab, so the common local-markdown path shows the tab with content already in place and skips the `Loading…` flash.
- Phase 2 foundations.
  - Sidebar file context menu: native Tauri menu on right-click of any markdown row with Open / Open in new tab / Duplicate / Copy relative path / Copy absolute path / Reveal in Finder (or Explorer / Show in Folder) / Rename… / Delete. New `reveal_in_file_manager` Rust command branches per platform (`open -R` on macOS, `explorer /select,…` on Windows, `xdg-open` on Linux). Added `openFileInNewTab` and `removePathReferences` editor-store actions, the `duplicateFile` helper that respects in-memory dirty buffers, and inline rename mode in `file-tree-node.tsx`.   - Gitignore-aware workspace: new `WorkspaceIgnore` matcher (`apps/desktop/src-tauri/src/ignore.rs`) walks every `.gitignore` under the workspace root and feeds them into a single `GitignoreBuilder`, with a hard-coded safety net for `node_modules` and `.git`. Loaded synchronously in `open_workspace` so the first `read_directory` already honors it; the watcher rebuilds the matcher whenever any `.gitignore` is created, modified, or removed and emits `fs:directory-changed` so the sidebar refreshes. Sidebar listing, recursive markdown probe, and watcher event filter all share the same matcher behind an `Arc<RwLock<…>>`.
  - Follow-up fixes on the ignore matcher: a nested `.gitignore` containing `*` no longer hides every file in the workspace (the root walk now scopes each `.gitignore` to its own subtree), and matcher (re)loads run off the IPC and watcher threads so an expensive `.gitignore` sweep can't stall document opens or file-system events.
